# Поэтапный план реализации: техпроцессы ЧПУ как Telegram Mini App

## 1. Что строим (кратко)

**Telegram Mini App** — обычное веб-приложение, которое открывается внутри Telegram. Пользователь заходит через бота (кнопка «Открыть»), видит интерфейс: детали → техпроцесс → справочник инструмента.

**Не входит в объём:**
- учёт смен, статусов, склада;
- режимы резания (обороты, подача, СОЖ — они в программе станка);
- патроны, центры, приспособления — только **кулачки** на токарном.

---

## 2. Концепция приложения (согласованная)

### Главный экран

1. **Поиск** — вводишь номер или название детали.
2. **Список деталей** — все детали участка, последние сверху или по алфавиту.
3. **Кнопки:**
   - **+ Новая деталь**
   - **Инструмент** — справочник всего, что есть на участке

### Разделы

#### Детали
- Номер / обозначение
- Название (втулка, фланец)
- Фотографии (чертеж, деталь, зажим — всё в фото)
- Привязанный техпроцесс

#### Техпроцессы
- Один техпроцесс на деталь
- **Установы** (I, II — переворот, перестановка)
- На каждом установа — **какие кулачки**
- **Операции** по порядку внутри установа

**На каждой операции:**
- Номер операции (010, 020…)
- Что делаем (черновое точение, расточка, резьба, фаска…)
- Инструмент (из справочника)
- Пластина (из справочника)
- Комментарий / подсказка

**Чего нет:** обороты, подача, СОЖ, глубина, припуск, позиция в револьвере — это в программе станка.

#### Инструмент (справочник)
Три подтипа в одном разделе:
- **Инструмент** — державки, резцы, свёрла, развёртки, метчики
- **Пластины** — CNMG 120408, DCMT 11T304 и т.д.
- **Кулачки** — кулачки 200 высота 15, мягкие губки

На каждой позиции: обозначение / название, примечание (опционально).

**Чего нет:** количество на складе, статусы «в использовании», «заказать».

#### История (опционально, на старте можно не делать)
- Когда последний раз правили техпроцесс
- Кто изменил и что добавил

## 3. Сервисы (кроме самого Telegram-бота)

| Сервис | Зачем | Обязательность |
|--------|-------|----------------|
| **Веб-сервер (фронтенд)** | Mini App — HTML/JS, открывается в Telegram | Обязательно |
| **API-сервер (бэкенд)** | Логика: детали, техпроцессы, инструмент, фото | Обязательно |
| **База данных** | Хранение деталей, операций, справочника | Обязательно |
| **Хранилище файлов** | Фотографии деталей | Обязательно |
| **HTTPS + домен** | Telegram открывает Mini App только по HTTPS | Обязательно |
| **Обратный прокси** | Раздача фронта, проксирование API, SSL | Обязательно |
| **Сервер/VPS или облако** | Где всё это крутится | Обязательно |

**Опционально (на старте не нужны):**

| Сервис | Когда понадобится |
|--------|-------------------|
| **Redis** | Кэш, очереди — если пользователей станет много |
| **CDN** | Быстрая раздача картинок из разных городов |
| **Резервное копирование БД** | Автобэкапы на проде |
| **Мониторинг** | Uptime, ошибки API |
| **CI/CD** | Автодеплой при пуше в репозиторий |

### Схема связей

```
Telegram (клиент)
    │
    ├─► Бот ──► кнопка «Открыть Mini App» (только запуск, не логика)
    │
    └─► Mini App (фронтенд, HTTPS)
            │
            ├─► API-сервер
            │       ├─► База данных
            │       └─► Хранилище файлов (фото)
            │
            └─► Telegram WebApp API (проверка, что зашёл из Telegram)
```

**Минимальный набор на старте:** один VPS, PostgreSQL, папка или MinIO для фото, nginx, API + фронт.

### Роль бота (не сервис, но важно)

Бот **не хранит данные** и **не дублирует API**. Только:
- команда `/start` или кнопка меню;
- кнопка `web_app` с URL Mini App;
- опционально: «Открыть последнюю деталь» — ссылка с `?part_id=`.

Всю логику делает Mini App + API.

## 4. Принципы модульности и изоляции

### 4.1. Слои (снизу вверх)

```
[База данных]
      ↑
[Репозитории]     — только чтение/запись в БД, без бизнес-логики
      ↑
[Сервисы]         — правила: «у детали один техпроцесс», порядок операций
      ↑
[API (роуты)]     — HTTP-запросы, вызов сервисов
      ↑
[Фронтенд-модули] — экраны по разделам: детали, техпроцесс, инструмент
```

**Правило:** модуль A не импортирует внутренности модуля B. Общение — через **контракты** (интерфейсы, типы данных, API).

### 4.2. Модули бэкенда (отдельные папки)

```
backend/
├── core/                 # конфиг, подключение к БД, общие ошибки
├── catalog/              # справочник: инструмент, пластины, кулачки
│   ├── models.py
│   ├── repository.py
│   ├── service.py
│   └── router.py
├── parts/                # детали + фото
│   ├── models.py
│   ├── repository.py
│   ├── service.py
│   ├── storage.py        # загрузка фото
│   └── router.py
├── tech_process/         # установы, операции
│   ├── models.py
│   ├── repository.py
│   ├── service.py
│   └── router.py
├── assembly/             # сводка «что нужно для детали»
│   └── service.py
└── main.py               # только подключение роутеров, без логики
```

- Удалить `assembly` — детали и техпроцесс работают сами.
- Добавить `history` — отдельная папка, не трогает `tech_process`.

### 4.3. Модули фронтенда

```
frontend/
├── app/                  # роутинг, оболочка Mini App
├── features/
│   ├── catalog/          # список и форма инструмента
│   ├── parts/            # список деталей, карточка, фото
│   └── tech-process/     # установы, операции, редактор
├── shared/
│   ├── api/              # клиент к API (по одному файлу на модуль API)
│   └── ui/               # кнопки, поля, списки
└── telegram/             # initData, тема Telegram — изолированно
```

### 4.4. Контракты между модулями

| Модуль | Отдаёт наружу | Не знает о |
|--------|---------------|------------|
| `catalog` | id, название, тип (tool/plate/jaw) | деталях, операциях |
| `parts` | id, номер, название, urls фото | как устроены операции |
| `tech_process` | установы, операции, ссылки на id из catalog | как рисуется UI |
| `assembly` | список уникальных инструментов/пластин/кулачков по part_id | только читает через API/сервисы |

Связь **операция → инструмент** — только `tool_id`, `plate_id` (внешние ключи). Справочник меняется — техпроцесс не переписываем.

## 5. Структура данных (минимум)

**Справочник (`catalog_items`):**
- `id`, `type` (tool | plate | jaw), `name`, `note` (опционально)

**Деталь (`parts`):**
- `id`, `number`, `title`, `created_at`

**Фото детали (`part_photos`):**
- `id`, `part_id`, `file_path` / `url`, `sort_order`

**Техпроцесс (`tech_processes`):**
- `id`, `part_id` (один к одному)

**Установ (`setups`):**
- `id`, `tech_process_id`, `order` (I, II…), `jaw_id` (кулачки из catalog)

**Операция (`operations`):**
- `id`, `setup_id`, `order`, `op_number` (010, 020), `title`, `tool_id`, `plate_id`, `comment`

## 6. Этапы реализации (каждый — отдельно)

### Этап 0. Каркас проекта
**Цель:** пустой, но запускаемый проект. Бизнес-логики нет.

**Делаем:**
- репозиторий: `backend/`, `frontend/`, `docker-compose.yml` (опционально);
- `core`: конфиг, подключение к PostgreSQL, health-check `GET /health`;
- фронт: заглушка «Техпроцессы ЧПУ», подключение Telegram WebApp SDK;
- nginx или встроенная раздача статики + прокси на API;
- HTTPS (Let's Encrypt) на домене.

**Проверка:** открывается Mini App из Telegram, `GET /health` → OK.

**Файлы:** `core/config.py`, `core/database.py`, `main.py`, `frontend/app/App.tsx`, `telegram/init.ts`.

**Не делаем:** детали, инструмент, техпроцесс.

---

### Этап 1. Справочник инструмента (`catalog`)
**Цель:** вести инструмент, пластины, кулачки. Ни от кого не зависит.

**Делаем:**
- таблица `catalog_items`;
- API: список, создание, изменение, удаление, поиск по имени, фильтр по `type`;
- экран «Инструмент»: три вкладки или фильтр — Инструмент / Пластины / Кулачки;
- форма: название, примечание.

**Проверка:** добавил «CNMG 120408» и «Кулачки высота 15» — они в списке, поиск находит.

**Модули:** только `backend/catalog/*`, `frontend/features/catalog/*`.

**Изоляция:** другие этапы не трогаем.

---

### Этап 2. Детали (`parts`)
**Цель:** список деталей, карточка, фото. Без техпроцесса.

**Делаем:**
- таблицы `parts`, `part_photos`;
- загрузка фото → хранилище файлов;
- API: CRUD деталей, upload/delete/reorder фото;
- экраны: список деталей, поиск по номеру/названию, карточка с галереей;
- кнопка «+ Новая деталь».

**Проверка:** создал «В-204 Втулка», загрузил 2 фото — всё отображается.

**Модули:** `backend/parts/*`, `frontend/features/parts/*`.

**Зависимости:** только Этап 0. От `catalog` и `tech_process` не зависит.

---

### Этап 3. Техпроцесс — каркас (`tech_process`)
**Цель:** один техпроцесс на деталь, установы с кулачками, без операций.

**Делаем:**
- таблицы `tech_processes`, `setups`;
- API: создать техпроцесс для `part_id`, добавить/изменить/удалить установ, выбрать `jaw_id` из catalog;
- на карточке детали — блок «Техпроцесс»: Установ I, Установ II, у каждого — кулачки.

**Проверка:** у В-204 два установа, у каждого указаны кулачки.

**Модули:** `backend/tech_process/*`, `frontend/features/tech-process/` (только установы).

**Зависимости:** Этап 1 (кулачки), Этап 2 (деталь). Операции — следующий этап.

---

### Этап 4. Операции в техпроцессе
**Цель:** шаги внутри установа.

**Делаем:**
- таблица `operations`;
- API: CRUD операций, смена порядка (вверх/вниз);
- поля: номер оп., что делаем, инструмент, пластина, комментарий;
- выбор инструмента и пластины — из `catalog` (типы tool и plate);
- UI: таблица операций внутри каждого установа.

**Проверка:** в Установе I четыре операции (010–040), у каждой свой инструмент и пластина.

**Модули:** расширяем только `tech_process` и `features/tech-process`.

**Изоляция:** операция хранит только `tool_id` / `plate_id`. Удаление позиции из справочника — отдельная политика в `catalog.service` (запрет или пометка «битая ссылка»).

---

### Этап 5. Сводка «Что нужно для детали» (`assembly`)
**Цель:** по детали автоматически список всего инструмента, пластин и кулачков.

**Делаем:**
- сервис `assembly`: по `part_id` обходит техпроцесс, собирает уникальные id из catalog;
- API: `GET /parts/{id}/required-items`;
- на карточке детали — блок «Нужно для изготовления».

**Проверка:** для В-204 список совпадает с операциями и кулачками установа.

**Модули:** только `backend/assembly/service.py` + один блок UI.

**Изоляция:** модуль только читает. Выключили — карточка детали и техпроцесс работают.

---

### Этап 6. Главный экран и навигация
**Цель:** поиск, список, две кнопки — как в концепции.

**Делаем:**
- главная: поиск, список деталей;
- кнопки: «+ Новая деталь», «Инструмент»;
- навигация: Детали | Инструмент;
- открытие детали → карточка (фото + техпроцесс + сводка).

**Проверка:** за 3 клика: нашёл деталь → вижу техпроцесс и список инструмента.

**Модули:** `frontend/app/` — только сборка готовых `features/*`.

---

### Этап 7. Telegram: авторизация и ограничение доступа
**Цель:** в API заходят только из Mini App.

**Делаем:**
- фронт передаёт `initData` от Telegram WebApp;
- бэкенд проверяет подпись (токен бота знает только сервер);
- опционально: белый список `telegram_user_id`;
- middleware в `core/`, роутеры не трогаем.

**Проверка:** запрос без initData → 401; из Telegram → OK.

**Изоляция:** вся Telegram-логика в `core/telegram_auth.py` и `frontend/telegram/`.

---

### Этап 8. Копирование техпроцесса
**Цель:** похожая деталь — не набивать с нуля.

**Делаем:**
- API: `POST /parts/{id}/copy-tech-process-from/{source_part_id}`;
- кнопка на карточке детали: «Скопировать с другой детали».

**Модули:** метод в `tech_process/service.py`, кнопка в UI.

---

### Этап 9. (Опционально) История изменений
**Цель:** когда и что меняли в техпроцессе.

**Делаем:**
- таблица `audit_log`;
- запись из `tech_process/service` при сохранении;
- экран «История» на карточке детали.

**Изоляция:** отдельный модуль `history/`. Нет модуля — всё работает без истории.

---

### Этап 10. Деплой и эксплуатация
**Цель:** стабильно работает на участке.

**Делаем:**
- docker-compose: api, db, minio (или volume), nginx;
- бэкап БД по расписанию;
- `.env` для секретов;
- инструкция: как привязать URL Mini App к боту в BotFather.

**Проверка:** с телефона в смене открыл бота → Mini App → нашёл деталь → техпроцесс на экране.

## 7. Порядок этапов

```
0 Каркас
   ↓
1 Справочник (catalog)     ← можно параллельно с 2 после 0
   ↓
2 Детали (parts)
   ↓
3 Техпроцесс: установы
   ↓
4 Операции
   ↓
5 Сводка (assembly)
   ↓
6 Главный экран
   ↓
7 Telegram auth
   ↓
8 Копирование ТП
   ↓
9 История (опционально)
   ↓
10 Деплой
```

**Минимально рабочий продукт для цеха:** этапы **0 → 1 → 2 → 3 → 4 → 6 → 7 → 10**.

## 8. API по модулям

| Модуль | Эндпоинты (пример) |
|--------|-------------------|
| `catalog` | `GET/POST /catalog`, `GET/PATCH/DELETE /catalog/{id}`, `?type=plate` |
| `parts` | `GET/POST /parts`, `GET/PATCH/DELETE /parts/{id}`, `POST /parts/{id}/photos` |
| `tech_process` | `GET/PUT /parts/{id}/tech-process`, `POST .../setups`, `POST .../operations`, `PATCH .../operations/reorder` |
| `assembly` | `GET /parts/{id}/required-items` |

Префиксы и версия (`/api/v1/`) — в `main.py` одной строкой на роутер.

## 9. Как не сломать при добавлении функции

| Действие | Где менять | Что не трогать |
|----------|------------|----------------|
| Новый тип в справочнике | `catalog`: enum `type` + вкладка UI | `tech_process`, `parts` |
| Поле «примечание к детали» | `parts.models` + форма | `catalog`, `assembly` |
| Экспорт техпроцесса в PDF | новый `export/router.py` | сервисы только читают через API |
| Второй станок / фрезер | новый модуль `machine_types` позже | текущие таблицы без поломок |

**Запрет:** один файл на 2000 строк и прямые SQL-запросы из роутов.

## 10. Стек

| Часть | Технология | Почему |
|-------|------------|--------|
| API | Python **FastAPI** | быстрые роутеры, понятное разбиение по модулям |
| БД | **PostgreSQL** | связи деталь–ТП–операции–справочник |
| Фото | **MinIO** (S3) или папка на диске | MinIO при росте |
| Фронт | **React + Vite** | стандарт для Mini App |
| Telegram | `@twa-dev/sdk` | обёртка WebApp |
| Прокси | **nginx** | HTTPS, статика, прокси `/api` |
| Сервер | VPS (1–2 GB RAM) | хватает на участок |

## 12. Резюме

1. **Сервисы:** фронт (Mini App), API, PostgreSQL, хранилище фото, HTTPS, nginx, VPS — бот отдельно, только открывает приложение.
2. **Этапы 0–10** — каждый закончен сам по себе, с проверкой.
3. **Модули:** `catalog`, `parts`, `tech_process`, `assembly` (+ опционально `history`) — отдельные папки на бэке и во фронте.
4. **Изоляция:** связи через id и HTTP-контракты; сводка и история только читают или пишут в свой угол.
5. **MVP для цеха:** каркас → справочник → детали с фото → техпроцесс → главный экран → Telegram auth → деплой.
